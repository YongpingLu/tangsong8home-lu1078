## 一、项目概览

「唐宋八大家」是一款面向古诗词学习与诵读的微信小程序，以"按出生年代 · 品读千古文章"为主线，把韩愈、柳宗元、欧阳修、苏洵、曾巩、王安石、苏轼、苏辙八位文学家的代表佳作、央视纪录片《宗师列传》入口、以及"我要跟读 + 回放自评"的诵读练习整合在一起。

**核心成果数字**

- 8 位文学家，按出生年代排序的时间线
- **188 首**诗词（诗 / 词 / 文），含原文、赏析、白话译文
- 央视《宗师列传·唐宋八大家》正片观看入口（含上下集）
- 原生录音跟读 + 单实例回放（支持暂停 / 续播 / 停止）
- 全文检索（按诗词名或内容模糊匹配）
- 工程已具备 GitHub 开源条件（README + .gitignore）

**一句话定位**：把"读、学、看、诵"四件事，塞进一个 2MB 主包的小程序里。

## 三、主要方法、碰到的问题与解决办法

### 3.1 贯穿全程的核心方法

1. **分包架构规避体积上限**：主包只留轻量页面（`index / person / poems`），详情页与音频放进分包 `packagePoem`，主包从 ~1.08MB 降到 0.07MB，顺利通过"主包 < 1.5M"检查。
2. **数据驱动、配置与代码分离**：诗人信息集中在 `data/authors.js`，诗词按作者分文件（`data/poems/*.js`），页面只做渲染 —— 增补 95 首诗词时无需改任何页面逻辑。
3. **合规优先的降级策略**：个人主体无法使用"微信同声传译"插件、原生录音又会被判定为声纹采集 → 主动把"范读"降级为"录音自评"，并用《声纹授权协议》页兜底合规。
4. **单实例 + 状态机（回放）**：用单个 `InnerAudioContext` 实例 + 三态 `playState` 管理回放，从根本上杜绝多实例声音重叠。
5. **事件用"函数调用"而非"属性赋值"**：微信原生 `RecorderManager` 的事件（`onStart/onStop/onError`）必须用函数写法，属性赋值在真机不触发。
6. **乐观 UI 更新**：回放开始即 `setData({playing:true})`，按钮即时切换，避免连点产生叠加。
7. **临时脚本注入、用完即删**：用 Python 脚本批量注入译文 / 新诗词，校验后删除脚本，保证仓库干净。
8. **交付必带 MD5 + 导入说明**：每个 zip 记 MD5，并要求"解压后打开直接含 `app.json` 的文件夹导入"，避免路径套层导致预览报错。

### 3.2 问题与解决办法（节选）

| 问题现象 | 根因 | 解决办法 |
|----------|------|----------|
| 审核被拒（声纹 + 范读无法体验） | 个人主体无录音类目资质，范读需插件不可用 | 先彻底移除语音；后恢复"录音自评"并加《声纹授权协议》页 |
| 真机"未捕获到录音" | `app.js` 隐私回调递归调用 `requirePrivacyAuthorize`，授权不生效 | 用户同意后直接 `resolve()`，不再递归 |
| 点了"我要跟读"却没出现"停止"按钮 | `requirePrivacyAuthorize` 嵌套在 `getSetting` 异步里，success 不回调 | 点击即时 `setData({reciting:true})` 并直接 `start()`，依赖全局 `onNeedPrivacyAuthorize` |
| 停止后仍提示"未捕获到录音" | `RecorderManager` 事件用 `recorder.onStop = cb` 属性赋值，真机不触发 | 改为 `recorder.onStop(cb)` 函数调用 |
| 预览报 `WXML file not found` | 用户解压多套一层目录 / 打开了旧项目 | 重打包（根目录直接 `app.json`）+ 教正确导入方式 |
| 视频链接外部打开 403 | `search.cctv.com` 搜索页无直链、防盗链 | 改用专辑接口 `getVideoListByAlbumIdNew` 抓正片 `tv.cctv.com/.../VIDE*.shtml` 直链（验证 200） |
| 单链接无法表达上下集 | 数据结构只支持单集 | 改为 `videos` 数组，视频页用"集数卡片 + 各自复制按钮" |
| 连点回放声音重叠、无停止 | 每次点击 `new InnerAudioContext()` 不回收 | 单实例 + `_destroyAudio()` 回收；按钮切换为"停止回放" |
| 回放不能从暂停处续播 | 用 `stop()`（清零进度）而非 `pause()` | 区分 `pause()/resume()`，状态升级为 `idle/playing/paused` |

## 五、与 WorkBuddy 的主要提示词与优化建议

### 5.1 几组有代表性的提示词

- **需求纠偏类**：「审核失败原因截图…… 我想保留我要跟读功能」—— 用截图 + 一句目标，把"先删后留"的矛盾一次说清。
- **真机反馈类**：「真机上测试，提示'未捕获到录音'」「没有【停止跟读】按钮」—— 直接贴现象，AI 据此反查根因。
- **数据爬取类**：「之前的作者介绍视频链接提示 403…… 从截图中的页面爬取每位作者的介绍视频，例如柳宗元的如下 <链接>」—— 给"错误现象 + 正确样例"，AI 自动换抓取路径。
- **交互打磨类**：「增加点击【回放自评】按钮播放时停止功能；有时点击出现声音重叠问题」「能否支持暂停回放，再次点击时从暂停时的位置开始播放」—— 把验收标准直接写进需求（"不可重叠""从暂停处续播"）。
- **开放设计类**：「把上下两集入口都显示出来（可设计两个复制按钮，上/下；或有更好的设计也行）」—— 把设计权交给 AI，得到比"两个按钮"更优的集数卡片方案。
- **总结类**：本需求（梳理成 Word / MD / PPT 分享文档）。

### 5.2 给"提示词"的优化建议（方法论沉淀）

1. **截图 + 文字 > 纯文字**：真机报错、预览异常、UI 错位，截图能让 AI 少猜 80%。
2. **业务约束前置**：一开始就说明"个人主体小程序""测试号"，可避免整条语音方案返工。
3. **把验收标准写进需求**：例如"连点不可重叠""从暂停处续播"，AI 会据此设计状态机而非补丁。
4. **敢用"更好的设计也行"**：给 AI 设计自由度，往往得到超出预期的方案（集数卡片即一例）。
5. **每次交付要求"MD5 + 导入说明"**：便于在多版本间回滚，也避免路径套层引发的预览报错。
6. **真机问题带上环境信息**：设备型号 / 微信版本 / 基础库版本，能大幅缩短排错路径。
7. **复杂任务分轮推进**：先定方案（Plan），再逐版交付，每版带校验（语法 / 链接 200 / 计数），降低单次出错面。

## 六、总结提升

**技术沉淀（可复用）**
- 微信小程序"主包 2MB"不是障碍，分包 + 懒加载 + 资源压缩三件套即可化解。
- 原生能力（录音 / 播放）的坑几乎都在"事件监听写法"与"权限时序"，按官方 API 函数调用、用 `onStart` 确认真正开始，可避开 90% 真机玄学。
- 回放场景的"单实例 + 状态机"是消除声音重叠的通用范式，可迁移到任何音频播放器。

**协作沉淀（可复用）**
- 把 AI 当"全程闭环的初级工程师"：写代码 → 跑校验 → 打包 → 记记忆，每轮都可独立验证。
- "截图 + 约束前置 + 验收标准 + 设计授权"四件套，是高质量提示词的公式。

**后续可提升方向**
- 若升级**企业主体**并申请"文娱-视频"类目，可把央视正片换成自有云存储直链，实现小程序内直接播放（而非复制链接）。
- 跟读可从"纯自评"升级为"波形对比 / 时长评分"等轻量反馈，提升练习价值。
- 苏轼苏辙实为**三集**（上 / 中 / 下），中集目前未入链，可作为下一步内容补全。
- 诗词可补充"按体裁筛选""收藏""打卡诵读"等，增强学习闭环。

## 七、彩蛋：开发过程中的有趣发现

- **苏轼苏辙的"三集之谜"**：央视《宗师列传》里"苏轼 苏辙"其实是上 / 中 / 下三集，但两人各占一集卡片，中集一度"消失"。最终保留上下两集入链，中集待补 —— 历史人物的合传，连纪录片都爱"拆成几集讲"。
- **央视接口的"套娃"抓取**：搜索页 403 打不开，但顺着专辑页 JS 找到内部 `getVideoListByAlbumIdNew` 接口，一次拉出 114 条分集，再按标题筛出每位作者的正片 —— 像剥洋葱一样拿到直链。
- **从 10MB 音频到"录自己"**：最初用 `edge-tts` 生成 750 句范读音频（10.2MB）想做"范读"，结果个人主体用不了插件、音频也成合规负担，最后反而回归"录自己、评自己"的朴素方案，更贴合"跟读"本意。
- **个人 vs 企业的"声纹两难"**：想做"范读 + 逐字评分"需要企业主体和声纹类目；个人主体只能"录音自评"。一条合规线，把产品形态分成了两个世界。
- **"与陆歆影一起"**：首页标语从"按出生年代 · 品读千古文章"变成"与陆歆影一起 · 按出生年代 · 品读千古文章"，淡金色署名让冷冰冰的工具多了一点"人味"。
